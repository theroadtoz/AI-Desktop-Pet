# 项目总体规划 — AI 对话虚拟桌宠

> **项目代号**：Desktop Pet
> **日期**：2026-05-31
> **版本**：v1
> **状态**：规划阶段

---

## 一句话概述

**低打扰的 Live2D AI 桌面伙伴** — 重点解决长期驻留时的资源占用、打扰程度、角色表现和 Windows 桌面融合，而不是追求功能数量最多。

---

## 为什么现在做

| 信号 | 说明 |
|------|------|
| **需求信号存在** | 桌宠和 AI 伴侣均有成熟产品，但两者结合的市场已经出现多款直接竞品，不能再称为空白 |
| **竞争进入早期密集期** | AI Desktop Pet、Deskie、Moe.AI、Project N.E.K.O. 等已经覆盖 Live2D/AI/语音/记忆中的多项能力 |
| **隐私是架构约束** | 本地模型可做到数据不离开设备；使用云端模型时必须明确披露请求发送给供应商 |
| **可争取的体验差异** | 低资源常驻、低打扰主动交互、窗口级行为和角色动作质量仍有验证空间 |
| **商业模式待验证** | 买断、DLC、BYOK 和可选云订阅均需通过价格实验和真实转化验证 |

## 产品定义

### 定位

```
纯视觉桌宠 (Shimeji)          AI 对话伴侣 (Character.AI)
       │                              │
       └────────── ★ 我们 ★ ──────────┘
                桌面存在感 + AI 对话
              + Live2D + 零门槛
```

### 目标用户

1. **二次元爱好者** — Shimeji 15 年社区的天然延伸，已有桌宠品类认知
2. **桌面工作者** — 程序员/设计师/学生，每天 PC 前 6+ 小时
3. **轻量 AI 用户** — 想和角色聊天，不需要 Replika 级别的情感深度
4. **隐私敏感者** — 不想聊天记录被云端训练

### 商业模式

**待验证假设：基础应用或角色包买断 + BYOK/本地模型 + 可选云服务。**

竞品上架价格只能作为价格锚点，不能证明销量、留存或盈利。`¥18-36`保留为首轮测试区间，不作为已验证结论。

## MVP 功能

| 优先级 | 功能 | 说明 |
|--------|------|------|
| **Must** | Live2D 桌面渲染 | 透明窗口 + 置顶 + 穿透 + 拖拽 + 多显示器 |
| **Must** | AI 对话 + 6 情绪表情 | 打字输入，自由文本生成，情感分类驱动表情 |
| **Must** | 快速首体验 | 安装后无需复杂环境配置即可看到和操作角色 |
| **Must** | 鼠标交互反馈 | 视线追踪、点击/戳反馈、摸头反应 |
| **Must** | 本地历史与透明隐私说明 | 历史默认本地；云模型请求明确告知会发送给供应商 |
| **Should** | 多模型切换 | DeepSeek / Ollama 本地 / 其他 OpenAI 兼容 API |
| **Should** | 窗口吸附 | 角色坐在活动窗口边缘（Desktop Mate 级体验） |
| **Should** | TTS + 口型同步 | 角色朗读回复，提升陪伴感 |
| **Should** | 模型导入 | 加载用户自己的 Live2D 模型 |
| **Should** | 低资源模式 | IDLE 时 5fps，CPU < 1% |
| **Won't** | 语音输入 / 长期记忆 / 屏幕感知 / 多角色 / 移动端 / Steam Workshop | v2 评估 |

## 技术架构

### v1 技术栈（锁定版本）

| 组件 | 技术 | 版本 | 锁定原因 |
|------|------|------|---------|
| 运行时 | Electron | 当前受支持稳定版 | Phase 0 验证最近三个受支持版本 |
| 渲染引擎 | Cubism Framework / 可选 PixiJS | 待验证 | 不为单个 Live2D 角色强制引入场景图 |
| Live2D 封装 | 官方 Cubism 5 SDK for Web | R5 系列 | 正式路线 |
| 遗留备选 | pixi-live2d-display | 仅原型 | 维护停滞，不作为长期核心依赖 |
| AI SDK | openai (npm) | latest | 兼容 DeepSeek API |
| AI 模型 | 云端兼容 API / 本地模型 | 待定 | 通过成本、质量和首体验验证后决定默认方案 |
| 存储 | better-sqlite3 | latest | 本地 SQLite |
| 加密 | electron-store + safeStorage | latest | DPAPI 加密 |
| 情感分类 | sentiment.js | latest | 本地轻量 fallback |

### 系统拓扑

```
Electron App（单体进程）
├── 角色窗口（BrowserWindow）
│   ├── transparent + alwaysOnTop('screen-saver')
│   ├── CSS pointer-events 穿透
│   └── PixiJS + Live2D 渲染
│
├── 聊天面板窗口（BrowserWindow）
│   ├── 消息列表 + 输入框
│   └── 流式显示 + 中断按钮
│
└── 主进程
    ├── ChatEngine（OpenAI SDK → DeepSeek）
    ├── EmotionClassifier（LLM 提取 + sentiment.js fallback）
    ├── CharacterFSM（7 状态：IDLE→LISTENING→THINKING→SPEAKING↻, SLEEPING, INTERRUPTED, ERROR）
    ├── ExpressionController + MotionController（分离控制）
    ├── SqliteStorage
    └── KeyStore（safeStorage 加密 BYOK API Key）
```

### 关键风险

| # | 风险 | 等级 | Phase 0 验证 |
|---|------|------|-------------|
| 1 | Electron 透明窗口、系统穿透和 GPU 兼容性 | **阻断** | P0-1 |
| 2 | Live2D WebGL context 丢失和 GPU 恢复 | **高** | P0-3 |
| 3 | 模型人设稳定性和供应商依赖 | **高** | 固定测试集 + Provider 替换 |
| 4 | 情绪检测无 fallback → 角色面瘫 | **高** | T-1.3 情感分类器 |
| 5 | 双窗口内存 300-450MB | **中** | T-2.6 低资源模式 |

## 竞品防御

| 最危险竞品 | 威胁 | 我们的护城河 |
|-----------|------|------------|
| Desktop Mate 加入 AI | 最高 — 已有桌面集成+Steam分发 | Live2D 二次元亲近感 + 本地优先 + 模型自由 |
| Open-LLM-VTuber 降低门槛 | 中 — 社区驱动速度有限 | App Store 级安装体验 vs "先装 Python" |
| Character.AI 推出桌面端 | 低 — 路径依赖太强 | 真正的窗口穿透/吸附他们短期做不到 |

**可积累能力**：角色资产和动作编排、低资源长期驻留、Windows 窗口交互细节、稳定性、品牌与社区。基础的 Live2D + AI + 透明窗口组合本身不是壁垒。

---

## 5 大死亡陷阱

| # | 陷阱 | 避法 |
|---|------|------|
| 1 | 追求 AI 对话深度 | Live2D 表情弥补文字不足，"不崩人设"即满分 |
| 2 | 默认订阅制 | 买断 ¥18-36，冲动消费零阻力 |
| 3 | 先做 UGC 平台 | v1 一个精角色 > 100 个粗角色 |
| 4 | 忽视常驻进程性能 | IDLE → 5fps，目标 CPU < 1% |
| 5 | 隐私只是 checkbox | 默认本地，云端是用户主动选择 |
