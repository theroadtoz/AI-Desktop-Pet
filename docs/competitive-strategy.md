## 一、修订后的定位

> 一个适合长期驻留的 Live2D AI 桌面伙伴：资源占用低、主动交互克制、角色动作自然，并能稳定融入 Windows 桌面工作流。

不再使用：

- 第一个
- 唯一
- 完全空白
- 零配置且完全本地
- 已验证买断制

## 二、目标用户假设

以下均为待用户研究验证的假设：

1. 每天使用 Windows 电脑 4 小时以上的桌面工作者。
2. 对 Live2D、动漫角色或桌宠有兴趣，但不愿配置复杂 AI 工具的用户。
3. 认为现有 AI 桌宠过重、过度主动或角色表现粗糙的用户。
4. 愿意在本地模型、云端模型质量和隐私之间做明确选择的用户。

## 三、核心价值排序

### 1. 长期驻留体验

首要指标不是功能数量，而是用户是否愿意持续开着：

- 空闲时不明显影响游戏、视频会议和开发工具。
- 不频繁弹窗或主动打断。
- 多显示器、DPI、睡眠唤醒后保持正常。
- 用户可以快速隐藏、静音、暂停或退出。

### 2. 角色表现

- 一个经过打磨的官方角色优先于大量粗糙角色。
- 表情、身体动作和对话状态分离控制。
- 动作应与上下文相符，而不是仅靠六个静态标签机械切换。
- 通过固定脚本和用户盲测评估自然度。

### 3. 桌面融合

- 系统级点击穿透与精确交互区域。
- 拖拽、多显示器边界和窗口位置恢复。
- 窗口吸附作为高价值实验，不在验证前承诺。
- 托盘、暂停和低干扰模式必须属于核心体验。

### 4. AI 对话

AI 对话以稳定、短而自然、符合角色为目标，不与 Character.AI 比拼平台级模型深度。

- Provider 可替换。
- 支持中断、重试和错误恢复。
- 云端模式明确披露数据流。
- 本地模式明确披露硬件和下载成本。

## 四、MVP 边界

### Must

1. 单个官方 Live2D 角色。
2. 透明置顶、系统级穿透、拖拽和托盘控制。
3. 视线、点击和三种以上待机反馈。
4. 文本对话和流式回复。
5. 4-6种基础表情，加独立身体动作。
6. 本地历史记录和准确的隐私说明。
7. 空闲降帧及可测量的资源基线。
8. GPU、睡眠和网络异常恢复。

### Prototype after core

1. 窗口吸附。
2. TTS 和口型同步。
3. 用户模型导入。
4. 本地 LLM 安装流程。
5. 简单长期记忆。

### Not in v1

1. ASR。
2. 屏幕感知和 Agent 工具调用。
3. 多角色同时运行。
4. 移动端同步。
5. UGC 商店或复杂脚本系统。

## 五、商业模式实验

不预先锁定单一模式。首轮验证四种组合：

| 方案 | 收入来源 | 主要风险 |
|---|---|---|
| 基础应用买断 | 一次性购买 | 难覆盖长期云推理成本 |
| 角色/DLC 买断 | 官方角色和语音包 | 需要稳定内容生产能力 |
| BYOK | 用户承担模型费用 | 首次配置门槛较高 |
| 可选云订阅 | 托管模型额度 | 用户是否愿意持续付费未知 |

推荐默认实验：

> 付费基础应用或角色包 + BYOK/本地模型 + 可选云额度。

价格通过真实试玩页、愿望单、购买意向和退款反馈验证，不能只依赖问卷自报。

## 六、隐私表述

允许使用：

- 对话历史默认保存在本机。
- 本地模型模式下，对话可以不离开设备。
- 云端模式会将当前请求发送给用户选择的模型供应商。

禁止使用：

- 所有数据永远不上传。
- 使用 DeepSeek API 但同时声称数据不出电脑。
- 未说明数据流的“完全隐私”。

## 七、成功标准

### 技术原型

- 当前受支持 Electron 版本在目标 Windows 环境稳定运行。
- 系统级穿透和角色交互可动态切换。
- 多显示器、DPI、睡眠唤醒和 GPU 异常可恢复。
- 空闲资源占用达到实测目标，而不是文档预设。

### 产品验证

- 用户能在 5 分钟内完成首次体验。
- 受试用户连续运行 30 分钟后认为不影响主要任务。
- 次日再次启动率和 7 日保留情况优于仅问“是否感兴趣”。
- 用户能准确理解本地和云端模式的隐私区别。
