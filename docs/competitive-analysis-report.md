## 摘要

本报告对"AI 驱动的 Live2D 桌面宠物"赛道进行系统性竞品分析，为毕业设计项目 **Desktop Pet** 提供设计依据。

**研究范围**：5 个直接竞品（AI Desktop Pet、Project N.E.K.O.、Desktop Mate、Open-LLM-VTuber、Deskie），覆盖商业产品和开源方案。

**核心发现**：

1. "Live2D + AI 对话 + 桌面常驻"已形成直接竞争，本项目不以"市场空白"为定位。
2. 现有竞品在**桌面融合质量**、**低打扰设计**、**AI 人设稳定性**三个维度上存在可测量的差异和改进空间。
3. 竞品普遍追求功能广度（多模态、Agent、Workshop），但在"长期桌面驻留的克制体验"上投入不足——这是本项目的论证切入点。

**本报告的三个分析维度**对应毕业设计的核心论证：

| 维度 | 测量方式 | 对应章节 |
|------|---------|---------|
| 桌面融合质量 | 7 项技术测试用例（实测 3 个竞品 + 本系统） | §4.1 |
| 低打扰设计 | 6 维客观设计分析（全部 5 个竞品 + 本系统） | §4.2 |
| AI 人设稳定性 | 消融实验（本系统有/无防护对比）+ 竞品定性讨论 | §4.3 |

## 一、市场格局

### 1.1 赛道定义

本报告研究的赛道为：**以 Live2D/3D 角色在 Windows 桌面上常驻渲染为核心交互形式、集成大语言模型实现对话能力的桌面应用**。

区别于：
- **纯 AI 对话伴侣**（Character.AI、Replika）：无桌面驻留和角色渲染
- **纯视觉桌宠**（Shimeji、Bongo Cat）：无 AI 对话能力
- **VTuber 直播工具**（VTube Studio、PrprLive）：面向直播而非桌面陪伴

### 1.2 市场参与者分类

```
                    桌面融合强
                         │
        Desktop Mate      │    【本项目】
        MateEngine        │    (目标：克制常驻体验)
                         │
                         │  AI Desktop Pet
        VPet             │  Project N.E.K.O.
        Shimeji          │  Deskie
                         │  Open-LLM-VTuber
                         │
    ──────────────────────────────────────── AI 对话强
                         │
        Bongo Cat        │  Character.AI
        Ropuka's Island  │  Replika
                         │
                    桌面融合弱
```

### 1.3 五类商业模式

截至 2026 年 6 月，市场已分化为五种清晰的商业模式：

| 模式 | 代表产品 | 核心逻辑 |
|------|---------|---------|
| **本地 AI 买断** | AI Desktop Pet（¥52） | 一次性付费，内置本地 LLM，无需 API Key |
| **免费 + 点数/内购** | Deskie（萌点制）、Molili AI Friends | 免费安装，AI 对话消耗虚拟货币 |
| **免费 + 角色 DLC** | Desktop Mate（$14.99/角色） | 本体免费，授权 IP 角色单独销售 |
| **完全免费开源** | Project N.E.K.O.（MIT）、Open-LLM-VTuber | 核心开源，社区驱动 |
| **低价买断 + BYOK** | AI Desk Pet（¥11） | 低门槛本体，用户自备 API Key |

### 2.1 AI Desktop Pet

| 属性 | 内容 |
|------|------|
| **开发商** | 42kami |
| **发布日期** | 2026-04-28 |
| **平台** | Windows（Steam） |
| **价格** | ¥52 买断（中国区），$12.99（美国区） |
| **技术栈** | Electron + Live2D Cubism SDK（官方授权）+ 内置本地 LLM + NVIDIA CUDA |
| **Steam 评价** | 82% 好评（约 50 条评价），SteamDB 评分 72.5% |

**产品定位**：Live2D 桌宠 + 本地 AI 角色扮演 + 视觉小说的"三位一体"离线体验。

**核心功能**：
- 完全离线运行：内置本地 LLM，无需 API Key、无需联网
- 语音交互：SenseVoice/Whisper 语音识别 + Kokoro TTS + 声音克隆
- 角色系统：兼容 SillyTavern 人设卡格式，内置 100+ 角色卡，支持世界书
- Steam 创意工坊：Live2D 模型、角色卡、世界书一键订阅
- 眼球追踪 + 表情/动作交互
- 30 种界面语言

**与本项目最直接的可比性**：同是 Live2D + AI 桌面常驻，是最接近的直接竞品。其"完全离线"策略验证了隐私优先的市场需求，但约 20GB 的安装体积和 8GB VRAM 的硬件要求限制了用户群。

**核心功能**：

## 四、三维度对比分析

> 这是毕业设计论证的核心章节。三个维度的测量方法详见 §4.1-4.3 引言。

### 4.1 Y1：桌面融合质量

**测量方法**：7 项技术测试用例，在统一测试环境（Windows 11, 1920×1080 @150% DPI, NVIDIA GTX 1660）下实测。

| # | 测试用例 | Desktop Mate | AI Desktop Pet | 本项目目标 |
|---|---------|:---:|:---:|:---:|
| 1 | **透明穿透**：角色区域点击穿过到下方窗口 | 通过 | 待实测 | 必须通过 |
| 2 | **拖拽移动**：拖拽后正确定位 | 通过 | 待实测 | 必须通过 |
| 3 | **多显示器**：拖到副屏渲染正常 | 通过 | 待实测 | 必须通过 |
| 4 | **DPI 切换**：125%/150%/200% 缩放清晰 | 通过 | 待实测 | 必须通过 |
| 5 | **睡眠恢复**：合盖再打开角色仍在 | 部分（GPU高占用） | 待实测 | 必须通过 |
| 6 | **置顶稳定性**：全屏应用不抢占置顶层 | 注意：已知问题 | 待实测 | 必须通过 |
| 7 | **窗口吸附**：是否吸附活动窗口边缘 | 支持 | 不支持 | v1 后期实验 |

**分析**：Desktop Mate 在测试 1-4、7 上表现最佳，是目前桌面融合的标杆。但其 GPU 占用高（用户反馈集成显卡达 70%）和置顶稳定性问题表明，即使是标杆产品在长期驻留场景下仍有不足。AI Desktop Pet 的实际表现需要通过 Phase 0 原型验证才能确定——这将成为本项目的第一个实验数据点。

### 4.2 Y2：低打扰设计

**测量方法**：6 维客观设计分析。基于产品公开文档、Steam 商店描述和功能列表进行分析，不依赖用户实验。

| # | 分析维度 | AI Desktop Pet | Project N.E.K.O. | Desktop Mate | Open-LLM-VTuber | Deskie | 本项目设计 |
|---|---------|:---:|:---:|:---:|:---:|:---:|:---:|
| 1 | **主动行为频率** | 低（需点击触发对话） | 中（Agent主动推送） | 极低（仅动画） | 中（可选主动说话） | 中（社区内容推送） | **低**（克制主动触发） |
| 2 | **主动行为层级** | 轻量（眼神追随） | 重量（弹窗+Agent操作） | 极轻（视线+动作） | 中（语音打断+主动话题） | 中（故事更新通知） | **轻量**（微动作优先） |
| 3 | **静默控制** | 未明确 | 隐私模式 | 不支持（无暂停） | 配置驱动 | 未明确 | **必须**（暂停/隐藏/静音 ≤2步） |
| 4 | **通知策略** | 无明显通知 | Agent结果通知 | 闹钟提醒 | 可选 | 社区更新通知 | **默认关闭**（可选开启） |
| 5 | **IDLE 资源占用** | 待实测 | 用户反馈高（游戏并行掉帧） | GPU 占用高（集成显卡70%） | 待实测 | 待实测 | **目标 CPU<1%** |
| 6 | **退出/恢复便捷性** | 托盘控制 | 托盘控制 | 不支持（无法隐藏到托盘） | 配置控制 | 托盘控制 | **必须**：托盘一键退出 |

**分析**：现有竞品普遍在"主动克制"上设计不足。Desktop Mate 因无 AI 对话而天然"低打扰"，但它牺牲了对话能力。Project N.E.K.O. 功能最全但打扰风险最高。**本项目的设计差异在于**：将"低打扰"作为一级设计目标而非副产物——主动行为有冷却、静默时段可配置、IDLE 帧率降级到 5fps。

### 4.3 Y3：AI 人设稳定性

**测量方法**：本项目采用消融实验（有防护 vs 无防护）+ 竞品定性讨论。原因是竞品使用不同底层 LLM（本地 Qwen vs 云端 API vs 混合），无法通过统一测试脚本做公平对比。

**消融实验设计**：

| 实验组 | 条件 | 测量指标 |
|--------|------|---------|
| **实验组 A**（完整防护） | 三层嵌套约束法 + 角色卡注入 + 滑动窗口摘要（50轮上限） | 人格一致性、知识边界、抗诱导、情绪合理性 |
| **对照组 B**（基础防护） | 仅 system prompt 描述人设，无额外机制 | 同上 |
| **对照组 C**（无防护） | 纯模型对话，无任何 system prompt | 同上 |

**测试集**：
1. **人格基准测试**（20 题）："你是谁？你的性格？你的喜好？" → 一致性评分
2. **知识边界测试**（10 题）：故意问角色不该知道的信息 → 是否正确表示"不知道"
3. **压力诱导测试**（15 题）：用户试图重新定义角色身份 → 是否坚持人设
4. **对话衰减测试**：连续 50 轮对话，对比前后 10 轮回复一致性
5. **情绪合理性测试**：20 轮情感对话 → 回复情绪是否匹配上下文

**竞品定性对比**：

| 维度 | AI Desktop Pet | Deskie | 本项目 |
|------|:---:|:---:|:---:|
| **人格系统** | 角色卡 + 世界书 | 角色创建工坊 | 三层嵌套约束法 |
| **记忆机制** | 兼容角色卡长记忆 | 长期记忆 | 滑动窗口 + 摘要 |
| **防崩塌机制** | 角色卡作为 context anchor | 不明确 | 每轮注入核心角色信息 + few-shot |
| **已知风险** | 约 20GB 体积限制分发 | 平台角色质量参差不齐 | 依赖 LLM 供应商稳定性 |

| 差异化维度 | 市场现状 | 本项目的论证切入点 |
|-----------|---------|------------------|
| **克制型常驻体验** | 竞品追求功能广度 | 将"低打扰"作为一级设计目标，主动行为有冷却/静默/降帧机制 |
| **桌面融合深度** | Desktop Mate 是标杆但有 GPU 和置顶问题 | 以 Electron 透明窗口实现更低资源占用的桌面融合 |
| **工程架构合理性** | 多数竞品未公开架构文档 | 三层嵌套约束法、表情/动作分离控制、Provider 抽象层等可论证的工程决策 |
| **隐私数据流透明** | 产品营销常模糊"本地"和"云端" | 明确区分本地/BYOK/托管云三种模式的数据流 |

### 5.3 定位陈述

> **Desktop Pet 是一个面向 Windows 桌面的 Live2D AI 桌面伙伴，通过在桌面融合质量、低打扰设计和 AI 人设稳定性三个维度上的工程优化，探索"长期桌面驻留的克制 AI 陪伴体验"这一设计空间。**

## 六、结论与设计启示

### 6.1 对系统设计的直接影响

| 竞品信号 | 设计决策 |
|---------|---------|
| AI Desktop Pet 的 20GB 本地模型证明"离线"不是零成本 | 本项目不将本地 LLM 作为 v1 默认路径；提供 BYOK 入口 |
| Desktop Mate 移除 Mod 引发差评潮 | v1 支持数据型角色包（不执行第三方代码）；许可和版本元数据必填 |
| Deskie 的点数焦虑 | 若采用云额度，显示可理解的剩余额度和成本预估 |
| Project N.E.K.O. 的并行性能问题 | v1 不做屏幕感知/ASR/Agent 工具调用；IDLE 降至 5fps |
| Open-LLM-VTuber 的流式中断机制 | 必须实现 AbortController 中断链；用户可随时取消 AI 生成 |
| Moe-AI 的卸载信任危机 | 必须提供托盘退出、自启动开关和完整卸载路径 |
